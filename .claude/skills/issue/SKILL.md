# Issue管理スキル

`docs/issues/` 配下でIssue(不具合・課題)を時系列に管理する。1 Issue = 1ファイル。
最上位は常に「ユーザー価値への影響」。影響 → 現象 → 原因 → 対応 の順に階層的に深めて書く。
過去の記録は書き換えず、追記とfrontmatter更新で現状を表現する。

## モード判定

引数・文脈から判定する:

- **新規作成**(デフォルト): 新しい問題・課題が報告された
- **更新**: 既存Issue(ID・タイトル・文脈で特定)への調査結果・対応内容の追記、ステータス変更
- **一覧**: 「一覧」「状況」など → 全Issueのfrontmatterを読み、ステータス別・深刻度順に要約して報告する(ファイルは作成しない)

## 新規作成の手順

1. 採番: `ls docs/issues/` で既存ファイル名の連番NNNの最大値を確認して +1(3桁ゼロ埋め、初回は `001`)
2. 日付: `date +%Y-%m-%d` で今日の日付を取得する(推測しない)
3. ファイル名: `YYYYMMDD-NNN-<slug>.md`(日付はハイフンなし8桁、slugは英小文字ケバブケース)
4. 同ディレクトリの [template.md](template.md) を読み、全セクションを埋めて作成する
   - **「1. ユーザー価値への影響」を最初に書く。** 誰にどう影響するか不明な場合は、推測で埋めずユーザーに確認する
   - 起票時点で判明していない項目は「未調査」「未定」と明記する(空欄・セクション削除は不可)
   - 経緯セクションに起票のエントリを書く
5. 関連するSpecがあれば frontmatter の `specs` に相互リンクし、Spec側の `issues` にも追記する

## 更新の手順

1. 対象ファイルを特定して読む
2. 該当セクション(調査ログ・根本原因・対応など)を現状に合わせて更新する
3. 「経緯」セクションの**末尾**に `### YYYY-MM-DD` 見出しで「何をやったか、何がわかったか、次に何をするか」を追記する(過去のエントリは編集禁止)
4. frontmatter の `status` と `updated` を更新する

## ルール

- 本文は日本語。コード・コマンド・エラーメッセージ・ログは原文のまま引用する
- ステータス遷移: `open` → `investigating` → `fixing` → `resolved` → `closed`(対応しない判断は `wontfix`)
- `resolved` にするのは修正を検証できたときのみ。検証方法と結果を経緯に書く
- 深刻度(severity): `critical`(主要機能が使えない) / `high`(主要な価値が損なわれる) / `medium`(回避策あり) / `low`(軽微)
- 調査で判明した事実と推測を区別して書く(推測には「仮説:」を付ける)
