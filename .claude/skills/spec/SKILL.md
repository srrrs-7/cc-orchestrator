# Spec管理スキル

`docs/specs/` 配下で機能仕様を時系列に管理する。1 Spec = 1ファイル。
最上位は常に「ユーザー価値」。価値 → 体験 → 要件 → 設計 → 実装計画 の順に階層的に深めて書く。
過去の記録は書き換えず、追記とfrontmatter更新で現状を表現する。

## モード判定

引数・文脈から判定する:

- **新規作成**(デフォルト): 新しい機能・仕様の文書化を求められた
- **更新**: 既存Spec(ID・タイトル・文脈で特定)への設計追記・ステータス変更・実装進捗の反映
- **一覧**: 「一覧」「状況」など → 全Specのfrontmatterを読み、ステータス別に要約して報告する(ファイルは作成しない)

## 新規作成の手順

1. 採番: `ls docs/specs/` で既存ファイル名の連番NNNの最大値を確認して +1(3桁ゼロ埋め、初回は `001`)
2. 日付: `date +%Y-%m-%d` で今日の日付を取得する(推測しない)
3. ファイル名: `YYYYMMDD-NNN-<slug>.md`(日付はハイフンなし8桁、slugは英小文字ケバブケース)
4. 同ディレクトリの [template.md](template.md) を読み、全セクションを埋めて作成する
   - **「1. ユーザー価値」を最初に書く。** ここが一文で書けない場合は、作る前にユーザーに「誰のための何か」を確認する
   - 未確定の項目は「未定」「検討中」と明記する(空欄・セクション削除は不可)
   - **テスト戦略・テスト基盤を変更する Spec** では、既存テストダブル(fake / stub / spy)の置換・削除を要件にする前に、各ダブルが検証する性質が実基盤(実 DB 等)で観測可能かで分類する。観測不能な seam(service→ポートの振り分け検証・compile-time の型証明・実装が到達し得ないエラー分岐)を検証するダブルは最初から例外カテゴリに置き、単純置換の対象にしない(残してよいダブルの類型は `.claude/rules/testing.md` と相互参照)
   - 経緯セクションに初版作成のエントリを書く
5. 関連するIssueから生まれたSpecなら frontmatter の `issues` に相互リンクし、Issue側の `specs` にも追記する

## 更新の手順

1. 対象ファイルを特定して読む
2. 該当セクション(要件・設計・実装計画など)を現状に合わせて更新する
3. 「経緯」セクションの**末尾**に `### YYYY-MM-DD` 見出しで「何を決めた/変えたか、なぜか」を追記する(過去のエントリは編集禁止)
4. frontmatter の `status` と `updated` を更新する
5. 大きな方針転換で旧Specを丸ごと置き換える場合は、新Specを作成して frontmatter の `supersedes` に旧IDを書き、旧Specの status を `superseded` にする

## ルール

- 本文は日本語。コード・API名・コマンドは原文のまま
- ステータス遷移: `draft` → `approved` → `in-progress` → `done`(中止は `dropped`、置換されたら `superseded`)
- `done` にするのは「価値の検証方法」に書いた条件を満たしたときのみ。確認結果を経緯に書く
- 実装がSpecから乖離したら、コードではなくSpecを現実に合わせて更新し、経緯に理由を残す
- 上位セクション(価値・体験)ほど安定させ、変更が頻発するのは下位(設計・実装計画)に留める。上位を変える場合は経緯に必ず理由を書く
